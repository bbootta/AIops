# GO_NO_GO_CHECKLIST

## 1. 목적

양적검증 팀 에이전트를 운영에 투입하기 전 필수 통제, 프로세스, 역할, 증적관리, 비상절차가 준비되었는지 확인한다.

## 2. Go/No-Go 원칙

- LLM 계산 방지 통제가 미흡하면 No-Go다.
- Amber라는 용어의 출력 판정 사용 방지 통제가 미흡하면 No-Go다.
- Gray 조건이 누락되면 No-Go다.
- Yellow / Red / Gray Action Notice 생성이 보장되지 않으면 No-Go다.
- 인간 최종 판단권과 공식 조직 승인권을 오인할 가능성이 있으면 No-Go다.
- 감사추적과 재현성이 불충분하면 No-Go다.

## 3. 체크리스트

|---|---|---|---|---|
| 정책 정합성 | Green/Yellow/Red/Gray 정의 승인 | 내부 정책 owner 검토 완료 |  |  |

| 계산엔진 연계 | LLM 직접 계산 차단 문구 | 시스템 프롬프트와 UAT에서 검증 |  |  |

| 보안 | 민감정보 처리 기준 | masking/권한/로그 정책 확인 |  |  |

## 4. No-Go 조건

다음 중 하나라도 해당하면 운영 투입을 보류한다.

- LLM이 직접 수치 계산 또는 임계값 산출을 수행할 수 있는 상태.
- 계산엔진 결과 없이 정량 결론을 출력하는 상태.
- 허용 라벨 외 판정이 출력되는 상태.
- Gray 강제 조건이 시스템 프롬프트, 판정 정책, UAT에 반영되지 않은 상태.
- Yellow / Red / Gray임에도 Action Notice가 생성되지 않는 상태.
- Green을 최종 승인처럼 표현하는 상태.
- 규제 변경을 자동 반영하는 상태.
- 감사추적에 입력자료, 정책 버전, 계산엔진 결과 ID, 인간 검토 결과가 남지 않는 상태.
- 운영자 교육과 비상중지 절차가 완료되지 않은 상태.

## 5. 최종 의사결정 기록 양식

```markdown
# Go/No-Go Decision Record
- decision_id:
- 검토일:
- 검토 조직:
- 참석자:
- UAT 결과 요약:
- 잔여 결함:
- 운영 리스크:
- 결정: Go | Conditional Go | No-Go
- 조건부 조치:
- 공식 승인자:
- 다음 검토일:
```

## 6. 운영상 가정

- Go 결정은 에이전트가 아니라 지정된 운영 승인 조직이 수행한다.
- Conditional Go는 제한된 범위, 기간, 대상 업무에 대해서만 허용한다.
