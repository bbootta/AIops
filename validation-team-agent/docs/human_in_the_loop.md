# Human-in-the-Loop Policy

본 하니스는 결정의 최종 책임을 **인간 검증자**에게 둔다. 다음 영역은 자동
실행이 금지되며, 반드시 인간 승인이 필요하다.

---

## 1. 절대적 인간 승인 영역

- 모형 승인 / 부적합 / 조건부 승인 의견의 최종 확정
- 검증보고서 / 검증의견서의 외부 제출본 확정
- 감독기관 제출용 수치, 표, 문안 확정
- 운영계 ECL 계산식 / PD·LGD·EAD 캘리브레이션 적용
- 스트레스 테스트 결과의 자본 계획 반영
- 트리거 임계값의 변경
- 검증 기준의 완화 또는 예외 적용

---

## 2. 자동 진행 가능 영역 (사후 검토 전제)

- 데이터 품질 점검 결과 산출
- 정량 지표 계산
- 보고서 초안 생성
- 점검표 자동 채움
- 실행 로그 기록
- 변경 이력 기록 (단, `status: validated` 전환은 인간 승인 후)

---

## 3. 중단 트리거 (Stop & Ask)

다음 상황에서 에이전트는 즉시 중단하고 사용자 확인을 요청한다.

- 운영계 접속 / 변경이 필요한 경우
- 민감정보가 포함된 원천 데이터가 필요한 경우
- 검증 기준이 불명확한 경우
- 분석 결과가 기존 검증 기준과 중대하게 충돌하는 경우
- 표본 수 부족으로 수치 해석이 불안정한 경우
- 모델 승인/부적합 의견 확정이 필요한 경우
- 외부 / 감독기관 제출 문안 확정이 필요한 경우

---

## 4. 검토 체크리스트 (인간 검증자용)

- [ ] 검증 의견의 근거가 충분한가
- [ ] 한계가 명시되어 있는가
- [ ] 추가 확인 사항이 누락되었는가
- [ ] 데이터·표본이 적절한가
- [ ] 모형 가정과 일관되는가
- [ ] 내부통제 / 문서화 요건을 충족하는가
- [ ] 외부 제출 형식 요건을 충족하는가
- [ ] 변경 이력이 매니페스트에 기록되었는가

---

## 5. 거버넌스 / 분기 보고 (CRO·MRMC·감사위원회)

본 하니스는 자체가 검증 의견에 영향을 주는 모형이므로 **2차 모형위험**으로
분류·관리되어야 한다. 운영 단계의 정책은 `docs/executive_summary.md` 참조.

분기 1회 다음을 의무 보고로 제안한다.

- `python -m tools.governance_kpi report` 출력
  - 매니페스트 status 분포 (proposed / applied / validated / rolled_back)
  - 자동 분류기 agreement_rate / top mismatch
  - 최근 audit run 의 executed / skipped / missing
  - policy_lint / sample_size_alignment pass 여부
- 병행 운영 기간(Phase 1) 동안 수기 vs 자동 검증 결과 차이
- Automation bias 모니터링: 검증자가 자동 산출물을 그대로 채택한 비율과
  정성 검토에서 추가된 finding 비율의 추이

---

## 6. 인력 운영 원칙

- 본 하니스 도입은 **검증팀 인력 감축 사유가 될 수 없다**.
- 자동화로 절약된 시간은 다음에 재배치되어야 한다.
  - 정성 검토 (가정 합리성, 사업 전략 정합성, 사용 부서 협의)
  - 챌린저 모형 분석
  - 신규 모형 사전 점검
  - 거시 시나리오 정성 보강
- 검증자 1인당 모형 담당 수를 늘리는 데 사용하지 않는다 (자동화 의존 가속 위험).
