# Project Overview

## 1. 검증팀 에이전트 개념

`validation-team-agent`는 은행 리스크관리 검증팀의 반복 업무를 표준화하고
재현 가능한 형태로 자동화하는 검증 보조 하니스다.

검증팀의 핵심 활동은 다음과 같이 구성된다.

- 데이터 품질 점검
- 모형 가정·방법론 검토
- 정량 성능 지표 산출 및 해석
- 시나리오 / 스트레스 분석 점검
- 내부통제 및 문서화 검토
- 검증보고서 / 검증의견서 작성
- 운영 모니터링 결과 점검

이 하니스는 위 활동을 **재현 가능한 컴포넌트**로 분해하고, 각 컴포넌트에 대해
실행 가능한 도구·정책·점검 미들웨어를 제공한다.

---

## 2. 왜 AHE(Agentic Harness Engineering)인가

전통적인 단일 LLM 워크플로우는 검증 업무에 다음과 같은 한계를 가진다.

- 산출물의 근거가 보존되지 않는다.
- 모형마다 적용 기준이 다른데 정책이 프롬프트 안에 섞여 있다.
- 변경 이력이 추적되지 않는다.
- 실패가 어디서 발생했는지 알 수 없다.

AHE는 다음 세 가지 관찰 가능성(observability)을 분리한다.

| 축 | 의미 | 본 프로젝트의 구현 |
|---|---|---|
| Component | 어떤 컴포넌트가 잘못 동작했는가 | `harness/`, `skills/`, `subagents/`, `tools/`, `middleware/` 분리 |
| Experience | 실행 흔적은 어땠는가 | `logs/`, `middleware/run_logger.py` |
| Decision | 어떤 변경이 어떤 근거로 이루어졌는가 | `harness/change_manifest.json` |

## 3. 적용 가능 업무

- **신용평가모형 검증**: KS, AUROC, Gini, PSI, 안정성, 챌린저 비교
- **PD/LGD/EAD 검증**: 캘리브레이션, 백테스팅, CDR/SDR 비교, 등급별 표본 적정성
- **IFRS 9 ECL 검증**: 스테이지 분류 일관성, 가중치 합산, 시나리오 가중치 정합성,
  forward-looking 변수 사용 적정성
- **거시 시나리오 모형 검증**: 변수 정상성, 다중공선성, 시차 변수, 잔차 진단
- **스트레스 테스트 검증**: 시나리오 서열 (base ≤ adverse ≤ severe), PD 배수 floor,
  이탈 케이스 점검
- **운영 모니터링**: PSI 추이, 등급 분포, 부도율 추이, 트리거 발동 점검
- **문서화 검증**: 보고서 필수 섹션 누락, 한계 명시 누락, 추가 확인사항 누락

---

## 4. 적용 제외 업무

- 운영계 시스템에 대한 직접 변경
- 감독기관 제출 문안의 최종 확정
- 모형 승인 / 부적합 의견의 최종 확정
- 법률·규정 유권해석
- 실제 고객 식별정보 처리

위 영역은 **인간 검증자**의 책임 범위이며, 본 하니스는 보조 입력만 제공한다.

---

## 5. 산출물의 위치

- 분석 결과 / 보고서 초안 → `reports/` (커밋 제외)
- 실행 로그 → `logs/` (커밋 제외)
- 반복 발견 사항 / 모형별 노트 → `memory/`
- 변경 이력 → `harness/change_manifest.json`
