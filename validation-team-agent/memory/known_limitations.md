# Known Limitations

본 하니스의 알려진 한계. 본 문서는 `memory/known_limitations.json`에서
자동 생성된다. 직접 편집하지 말고 `python -m tools.limitations sync`
명령을 사용하라. 인간 검증자가 운영 시 반드시 인지해야 한다.

## 1. 통계적 한계
- KS / AUROC / Gini 등 변별력 지표는 표본이 작거나 클래스 불균형이 심하면 분산이 커진다. 본 하니스는 신뢰구간을 자동 산출하지 않으며, 작은 표본에서는 결과 해석에 주의가 필요하다.
- PSI는 bin 정의에 민감하다. 본 하니스는 분위수 기반 bin (default 10)을 사용한다. 도메인 정의 기반 bin을 사용해야 하는 모형은 호출자가 별도 처리한다.
- 등급 단조성 점검은 단순 비교로 시뮬레이션된다. 통계적 유의성 검정은 미구현이다.

## 2. 데이터 한계
- 본 하니스는 운영 DB에 직접 접근하지 않는다. 따라서 운영 원천과 검증 추출 간 스키마 차이는 사용자가 사전 정합화한다.
- 민감정보 패턴 탐지(data_safety_guard)는 정규식 기반이며 false positive / false negative가 있을 수 있다.

## 3. 방법론 한계
- 회귀 진단(regression_diagnostics)은 OLS에 한정된다. 비선형/패널/혼합 모형은 별도 라이브러리 사용이 필요하다.
- 시계열 정상성 검정은 ADF/KPSS 만 지원한다 (CHG-0003 이후). 구조변화 검정은 호출자가 별도로 수행한다.

## 4. 문서/내부통제 한계
- output_completeness_guard는 형식 점검에 한정된다. 의견의 실질적 적정성은 인간 검증자가 판단한다.
- permission_guard는 패턴 기반이며 우회 가능한 형태가 존재할 수 있다. 통제는 CLAUDE.md의 권한 제한과 함께 적용되어야 한다.

## 5. 결정 한계
- 본 하니스는 모형 승인 / 부적합 의견을 자동 확정하지 않는다.
- 외부 제출본 / 감독기관 대응 문안 확정은 인간 승인 필수.

## 6. 거버넌스 / 2차 모형위험
- 본 하니스 자체가 검증 의견에 영향을 주는 모형이므로 MRMC(모형위험관리위원회)의 분류·승인을 받아야 한다. 분류 제안: Tier 2 보조 모형.
- Automation bias 위험: 자동 산출물에 검증자가 과의존하여 정성 판단을 생략할 수 있다. 도입 초기 6개월은 수기 검증과 병행 운영을 권고한다.
- 본 하니스 도입을 검증팀 인력 감축 사유로 사용해서는 안 된다. 자동화로 절약된 시간은 정성 검토·챌린저 모형·신규 모형 사전 점검으로 재배치되어야 한다.
- matrix/policy 파일의 schema 통과는 형식 적합성일 뿐 정책 수치 자체의 적정성을 보장하지 않는다. 임계 변경은 반드시 매니페스트에 근거를 기록한다.
- promote-if-passing CLI의 --i-am-human 플래그는 감사 추적 신호일 뿐 실질 권한 분리는 docs/executive_summary.md의 권한 매트릭스로 운영해야 한다.
